# /review — External Benchmark Review & Analysis

검토/분석 요청 시 실제 서비스 중인 앱/웹을 근거로 분석하고, 결과만 보고한다.

**핵심 원칙**:
1. **실제 서비스 벤치마크 필수** — 주관적 의견이 아닌, 실제 프로덕션 앱에서 근거를 찾는다
2. **READ-ONLY** — 파일 수정 절대 금지. 분석/검토 결과만 보고한다
3. **프로젝트 특성 기반** — 현재 프로젝트의 도메인에 맞는 경쟁/유사 서비스를 참고한다

---

## Step 0: 금지 사항 (CRITICAL)

- **파일 수정 금지** — Edit, Write, NotebookEdit 도구 사용 금지
- **코드 자동 수정 제안 금지** — "수정해드릴까요?" 류의 제안 금지
- **주관적 판단 금지** — "이렇게 하는 게 좋을 것 같습니다" 대신 "TripAdvisor는 이렇게 한다" 식으로 근거 제시

---

## Step 2: 분석 대상 파악

사용자의 요청에서 분석 대상을 파악한다:

| 요청 유형 | 대상 |
|----------|------|
| "/review" (범위 없음) | 마지막으로 작업한 파일/기능 |
| "/review [파일/기능 명시]" | 명시된 파일 또는 기능 |
| "이 로직 검토해줘" | 현재 맥락의 로직 |
| "이 UI 분석해줘" | 현재 맥락의 UI 패턴 |
| "전체 UX 검토해줘" | 앱 전반의 UX 플로우 |

## Step 3: 분석 수행

### 3A. UI/UX 분석 시

실제 서비스에서 동일한 기능의 UI 패턴을 조사하고 비교한다.

```
분석 항목:
□ 레이아웃 패턴 — 해당 기능을 실제 앱들은 어떻게 배치하는가?
□ 인터랙션 패턴 — 터치, 스와이프, 롱프레스 등 어떤 인터랙션을 쓰는가?
□ 정보 계층 — 어떤 정보를 먼저/크게/강조해서 보여주는가?
□ 네비게이션 — 해당 기능으로 어떻게 진입하고 빠져나가는가?
□ 상태 피드백 — 로딩, 에러, 빈 상태를 어떻게 처리하는가?
□ 마이크로 인터랙션 — 애니메이션, 햅틱, 토스트 등
```

### 3B. 로직/아키텍처 분석 시

실제 서비스의 공개된 기술 블로그, 오픈소스, 컨퍼런스 발표를 근거로 분석한다.

```
분석 항목:
□ 데이터 플로우 — 상태 관리, 캐싱 전략, 동기화 패턴
□ 에러 처리 — 에러 복구, 재시도, 폴백 전략
□ 성능 — 렌더링 최적화, 네트워크 최적화, 메모리 관리
□ 오프라인 — 오프라인 지원 범위, 동기화 전략
□ 보안 — 인증, 권한 관리, 데이터 보호
□ 확장성 — 데이터 증가, 사용자 증가 대응
```

### 3C. 기능 분석 시

경쟁 서비스의 동일 기능과 비교하여 완성도를 평가한다.

```
분석 항목:
□ 기능 범위 — 경쟁 서비스 대비 누락된 기능은?
□ 엣지 케이스 — 경쟁 서비스가 처리하는데 우리가 놓친 케이스는?
□ 사용자 플로우 — 동일 작업 완료까지 단계 수 비교
□ 차별점 — 경쟁 서비스에 없는 우리만의 강점은?
```

## Step 4: 근거 제시 방식

모든 분석 결과에는 반드시 실제 서비스의 근거를 포함한다.

### 근거 포맷

```markdown
**현재 구현**: [현재 앱의 패턴 설명]
**벤치마크**:
- **TripAdvisor**: [TripAdvisor가 같은 기능을 어떻게 구현하는지]
- **트리플**: [트리플이 같은 기능을 어떻게 구현하는지]
**분석**: [근거 기반 비교 분석]
```

### 근거 수집 방법 (우선순위)

1. **WebSearch** — 해당 서비스의 UX 패턴, 기술 블로그, 디자인 가이드 검색
2. **공식 디자인 가이드** — Apple HIG, Material Design, 각 서비스의 디자인 시스템
3. **기술 블로그** — 해당 서비스의 엔지니어링 블로그 (예: Airbnb Engineering)
4. **오픈소스 참고** — 유사 기능의 오픈소스 구현체
5. **컨퍼런스 발표** — 해당 서비스의 기술 발표 자료

## Scope Options

- `/review` — 마지막 작업 대상 검토
- `/review ui [기능]` — 특정 기능의 UI/UX 검토
- `/review logic [파일/기능]` — 특정 로직/아키텍처 검토
- `/review feature [기능]` — 기능 완성도 검토
- `/review ux-flow [플로우]` — 사용자 플로우 전체 검토
- `검토해줘`, `분석해줘`, `리뷰해줘` — 맥락에서 대상 자동 파악
